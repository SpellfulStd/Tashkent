# ADR-0006: Модель «игрок ↔ аккаунт», гости, объединение, видимость

**Статус:** принято · Этап 2–3

## Контекст
Игрок (участник партии) и аккаунт (личность Keycloak) — разные сущности. Нужно: регистрировать новых, играть с «гостями» без аккаунта, связывать существующих игроков с аккаунтами, не плодить дубликатов, и ограничить видимость чужих игр.

## Решение
- **Игрок** имеет опциональную привязку `account_sub`. Гость — без привязки; за него отмечает любой авторизованный участник состава.
- **Авто-игрок:** при первом входе пользователь автоматически становится игроком.
- **Объединение:** непривязанного игрока (гостя) можно слить в свой профиль — перенос состава, событий, счёта; дубликат удаляется (транзакционно).
- **Автозаход:** при входе, если игрок пользователя уже в составе активной игры/серии — попадает в неё; иначе видит в режиме просмотра.
- **Видимость:** `GET /api/games` отдаёт только игры со своим участием; владелец (`username==='spellful'`, `isAdmin`) видит все.

## Последствия
- (+) Гибкость: смешанные составы (аккаунты + гости), естественная миграция «гость → профиль», приватность истории.
- (−) Простая модель «админ = конкретный username»; для роста нужна ролевая модель (роли Keycloak).
- (−) Объединение односторонне переносит историю; обратной операции нет (откат — восстановлением из бэкапа).
