# ADR-0001: PostgreSQL как хранилище данных

**Статус:** принято · Этап 2

## Контекст
Прототип хранил данные в JSON-файлах — это не выдерживает многопользовательский real-time, конкурентные записи и связность (игроки ↔ игры ↔ события ↔ аккаунты). Параллельно вводилась авторизация через Keycloak, которому тоже нужна БД. Владелец делегировал выбор («выбери, что лучше»): Postgres или Mongo.

## Решение
**PostgreSQL.** Данные реляционные (M:N составы, ссылки событий на игроков, серии), нужны транзакции на запись партии. Keycloak использует тот же экземпляр (отдельная БД `keycloak`) — меньше инфраструктуры. События — нормализованная append-only таблица (удобно для real-time и аудита), `final_scores` — JSONB (компромисс гибкость/простота).

## Последствия
- (+) Целостность, транзакции, единый сервер БД для приложения и Keycloak; сессии тоже в Postgres.
- (+) Миграция JSON→Postgres выполнена без потерь (сверено), с защитой FK.
- (−) Появилась эксплуатация БД: бэкап (`pg_backup.sh` + systemd-таймер), volume `pgdata`.
- Mongo отклонён: данные реляционные, а Keycloak с Postgres интегрируется штатно.
