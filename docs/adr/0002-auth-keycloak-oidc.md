# ADR-0002: Авторизация через Keycloak (OIDC)

**Статус:** принято · Этап 2

## Контекст
Приложение становилось многопользовательским: нужны вход, регистрация, идентичность игроков. Владелец явно выбрал Keycloak (с прицелом на единый вход по поддоменам в будущем).

## Решение
**Keycloak (OIDC, authorization code flow).** Realm `spellful`, confidential-клиент `tashkent`, регистрация включена. Приложение — OIDC-клиент (`openid-client`), сессии в Postgres (`connect-pg-simple`), cookie `secure/httpOnly/sameSite=lax`. Keycloak за nginx на `auth.spellful.site`; приложение ходит к нему через `host-gateway` с валидным TLS. Кастомная тема `spellful` под дизайн продукта. Новый пользователь при входе автоматически становится игроком; промежуточные страницы `/login`,`/register` ведут на нужные эндпоинты Keycloak (`/registrations` — сразу на регистрацию).

## Последствия
- (+) Стандартный, расширяемый IdP; SSO-потенциал на другие поддомены; формы стилизованы.
- (−) Keycloak — тяжёлый JVM-сервис (heap ограничен 512 МБ; добавлен swap во избежание OOM).
- (−) Отдельный мастер-админ Keycloak и поддомен/сертификат под него.
- Замечание безопасности: пароль владельца `spellful` слабый — критично при включённом админ-агенте (см. ADR-0004).
