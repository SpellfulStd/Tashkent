# ADR-0004: Админ-чат как автономный codex-агент (паттерн хост-воркера)

**Статус:** принято · Этап 3 · **несёт значимый риск безопасности (осознанно)**

## Контекст
Владелец захотел давать задания на доработку продукта прямо из интерфейса сайта, «в чат», и чтобы изменения сами попадали в репозиторий. Эта сессия Claude Code не подключена к сайту постоянно. Выбран вариант «живой агент»: форма запускает Codex на репозитории, тот сам вносит правки и коммитит.

## Решение
**Очередь + хост-воркер.** Приложение (в контейнере) не может запускать хостовый `codex` и перезапускать контейнеры, поэтому:
`POST /api/admin/chat` (только `spellful`) → `admin_tasks` → systemd-воркер `tashkent-codex-worker` на хосте → `codex exec -C /opt/tashkent` (sandbox `workspace-write`, системный контекст из `tashkent_agent_context.txt`) → `git commit/push` → при правках бэкенда `docker restart` → статус/вывод в `admin_tasks`. Изображения из чата сохраняются в проект и читаются codex с диска.

## Последствия
- (+) Самодорабатываемый продукт на естественном языке; каждое изменение — отдельный коммит (откатываемо); агент изолирован каталогом проекта.
- (−) **Фактически RCE и git-push для любого, кто войдёт под `spellful`.** Пароль `spellful` слабый → настоятельно рекомендована смена на сильный (на момент написания — не выполнена). Это главный незакрытый риск.
- (−) Автоизменения прод напрямую; гарантий корректности нет (откат — через git).
- Безопасные альтернативы (очередь задач без автозапуска; ассистент-обсуждение без правок) отклонены по явному выбору владельца в пользу автономности.
